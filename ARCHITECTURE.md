# HAIR CLUB — AI-агент для салона красоты
## Полная документация проекта

Этот документ объясняет, что за система, из каких частей состоит, зачем нужна
каждая часть и как они связаны между собой. Написан для того, чтобы можно было
понять весь проект целиком, а также передать этот файл в Claude Code для
дальнейшей разработки — там будет достаточно контекста, чтобы продолжить
работу без потери понимания архитектуры.

---

## 1. Что это за система, в двух словах

AI-агент, который вместо администратора общается с клиентками салона в Instagram
(через ManyChat) или Telegram, отвечает на вопросы по ценам/услугам, показывает
фото примеров работ, и **сам записывает клиентку на реально свободное время** —
без стороннего сервиса типа Bumpix, с собственной системой бронирования.

Плюс отдельная **PWA-панель**, где владелец салона сам:
- вносит услуги, цены, сотрудников, фото работ
- задаёт "характер" общения агента (тон общения)
- блокирует даты/добавляет акции и произвольные инструкции агенту
- смотрит все диалоги и может в любой момент перехватить разговор и писать клиентке сам

Плюс отдельный **публичный сайт бронирования** — клиентка может записаться и
напрямую через сайт, не только через переписку с ботом.

## 3. Структура файлов проекта

```

    └── index.html                # публичная страница бронирования для клиентов
```

## 4. База данных — что за что отвечает

| Таблица | Назначение | Кто пишет | Кто читает |
|---|---|---|---|
| `salons` | Профиль салона: название, адрес, часы, **тон общения агента** | Владелец через PWA | Агент (при сборке промпта) |
| `services` | Услуги: название, цена, длительность, описание | Владелец через PWA | Агент, сайт бронирования |
| `service_photos` | Фото примеров работ с подписью (используется, чтобы агент понимал, какое фото к какому вопросу подходит) | Владелец через PWA | Агент (function calling `attach_photo`) |
| `employees` | Сотрудники, график работы | Владелец через PWA | Агент, расчёт слотов, сайт бронирования |
| `employee_services` | Связь: какой сотрудник что умеет | Владелец через PWA | Сайт бронирования (фильтр мастеров под услугу) |
| `knowledge_base` | База знаний / FAQ (например, из книги про типы волос, уход) | Владелец через PWA | Агент (только когда клиент явно спрашивает) |
| `rules_overrides` | Динамические правила: блокировка дат, акции, произвольные инструкции с датами действия | Владелец через PWA | Агент (всегда подтягивает активные на сегодня) |
| `conversations` | Диалоги с клиентами, статус (active / escalated / owner_active) | Автоматически | PWA (список диалогов), агент |
| `messages` | Сообщения внутри диалога (от клиента / агента / владельца) | Агент, владелец (при перехвате) | Агент (память диалога), PWA |
| `bookings` | Записи клиентов на услуги | Агент (`create_booking`), сайт бронирования | PWA, booking-slots.js (проверка занятости) |
| `escalations` | Лог случаев, когда агент передал диалог человеку | Агент (`escalate_to_owner`) | PWA (бейджик "нужно внимание") |

## 5. Как работает диалог с клиентом (пошагово)

1. Клиентка пишет в Instagram → ManyChat получает сообщение → ManyChat дёргает
   наш Worker (`index.js`) через "External Request" с телом `{channel_id, message}`
2. `index.js` находит или создаёт диалог (`conversations`) для этого клиента
3. Если владелец **сейчас сам** ведёт этот диалог (`status = owner_active`) —
   агент не вмешивается, сообщение просто сохраняется в историю
4. Иначе — вызывается `agent.js`:
   - `retrieval.js` ищет 2-3 самые релевантные услуги, 1-2 фото, 1-2 FAQ-записи
     по смыслу вопроса клиента (без загрузки в промпт вообще всей базы)
   - подтягиваются **все** активные правила от владельца (их обычно мало)
   - собирается компактный системный промпт из всего этого
   - Claude либо отвечает текстом, либо вызывает функцию:
     - `attach_photo` — показать фото примера работы
     - `get_available_slots` — посмотреть свободное время у мастера
     - `create_booking` — создать запись (только после подтверждения слота и данных клиента)
     - `escalate_to_owner` — передать диалог человеку (конфликт, неуверенность)
5. Ответ агента сохраняется в `messages` и уходит обратно клиенту через ManyChat

## 6. Как работает сайт бронирования (для клиента напрямую, без чата)

1. Клиентка заходит на `hairclub.pl` (или похожий URL)
2. Выбирает услугу → видит только тех мастеров, кто её делает (`employee_services`)
3. Выбирает дату → `booking-slots.js` считает свободные слоты (график мастера
   минус уже существующие записи на этот день)
4. Выбирает время → вводит имя и телефон → нажимает "подтвердить"
5. `public-routes.js` ещё раз проверяет, что слот не заняли за это время кто-то
   другой (защита от двойной записи), и создаёт запись в `bookings`

## 7. Как работает панель владельца (PWA)

Шесть разделов, каждый — это просто удобный интерфейс поверх соответствующей
таблицы базы данных:

- **Услуги** → таблица `services` + `service_photos`
- **Мастера** → таблица `employees` + `employee_services`
- **Правила** → таблица `rules_overrides` (блокировка дат / акции / инструкции)
- **База знаний** → таблица `knowledge_base`
- **Диалоги** → чтение `conversations`/`messages`, и кнопка "написать самой" —
  добавляет сообщение с `sender='owner'` и переводит диалог в статус `owner_active`
- **Профиль** → таблица `salons`, включая поле **"тон общения"**, которое
  напрямую подставляется в промпт агента при каждом ответе

Панель защищена токеном (`ADMIN_TOKEN`) — без него никто чужой не сможет
менять данные, даже зная адрес сайта.

## 8. Почему архитектура именно такая (ключевые решения и зачем)

- **Поиск по смыслу (retrieval) вместо "всё в промпт"** — чтобы агент не путался
  и не "тупил" при росте базы услуг/FAQ, и чтобы не росли расходы на токены.
- **Function calling (агент явно вызывает действия)** — разделяет "агент решает,
  что сказать" и "агент решает, что сделать" (записать, показать фото, эскалировать).
  Это снижает риск ошибок в реальных действиях (например, ошибочная запись).
- **Эскалация к человеку** — агент не должен "выкручиваться" в сложной ситуации;
  у него есть чёткий путь передать разговор владельцу.
- **Своя система бронирования вместо Bumpix** — убирает зависимость от чужого
  сервиса без API, устраняет риск хрупкой автоматизации и делает всю систему
  полностью подконтрольной.
- **ManyChat вместо прямой интеграции с Meta** — ManyChat уже прошёл верификацию
  Meta как платформа, поэтому салону не нужно проходить долгий Business
  Verification самостоятельно — только лёгкая OAuth-авторизация.
- **Cloudflare Workers + D1** — бесплатный старт без обязательной привязки карты
  (в отличие от Firebase), SQL-база (не пришлось переписывать структуру данных
  под NoSQL), и инфраструктура, уже знакомая владельцу по прошлому проекту.

## 9. Что ещё не реализовано (открытые задачи)

- Дизайн интерфейса (PWA и сайт бронирования) — сейчас базовая вёрстка с
  цветами из логотипа (золото/охра). Требуется доработка под фирменный стиль:
  светлый, белый + золото + акценты зелени (растительность), более изящная
  типографика. Ссылка на логотип "HAIR CLUB" и референсы интерьера будут
  переданы отдельно — доработать после запуска базовой функциональности.
- Загрузка фото услуг как файлов (сейчас только текстовое поле с URL —
  нужно подключить Cloudflare Images или R2 для реальной загрузки картинок)
- UI для связи "мастер ↔ какие услуги делает" в самой панели (таблица в базе
  есть, интерфейса с чекбоксами пока нет)
- Адаптер под формат вебхука ManyChat (нужно сверить точный формат, который
  присылает ManyChat External Request, и при необходимости написать тонкую
  прослойку-переводчик под наш единый формат `{channel_id, message}`)
- Уведомления клиенту о записи (email/SMS) — пока запись просто сохраняется
  в базу без автоматического подтверждения на телефон
- Контроль "человечности" речи (запрещённые слова, минимум смайликов, анти-AI
  стиль речи) — уже добавлено в системный промпт агента и в панель профиля,
  но стоит проверить на реальных диалогах после запуска и доработать при
  необходимости

## 10. Что нужно зарегистрировать/настроить для запуска (кратко, подробности — в README.md)

1. Аккаунт Anthropic (API-ключ) — уже есть с прошлого проекта
2. Аккаунт Cloudflare — уже есть с прошлого проекта
3. Домен (например `hairclub.pl`) — привязать к Cloudflare
4. Создать базу D1, залить `schema.sql` и `seed.sql`
5. Задать секреты Worker'а: `ANTHROPIC_API_KEY`, `ADMIN_TOKEN`
6. Задеплоить Worker (`wrangler deploy`)
7. Разместить `pwa/` и `booking-site/` как статику (Cloudflare Pages) на
   поддоменах, вписать в них реальный адрес Worker'а вместо заглушки
8. Оформить ManyChat (Pro-план) и подключить через "External Request" к
   адресу нашего Worker'а — когда будет готово подключение Instagram
